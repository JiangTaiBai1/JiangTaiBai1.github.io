# TypeScript 函数类型注解：`typeof` 与参数类型，为什么封装函数常用第一种？

在封装一个现有函数（比如 Taro 的 `downloadFile`）时，你可能会看到两种常见的 TypeScript 类型写法：

### 第一种：只约束参数类型

```typescript
export const downloadFile = async (option: Taro.downloadFile.Option) => {
  // 实现...
}
```

### 第二种：完全复制原函数类型

```typescript
export const downloadFile: typeof Taro.downloadFile = async option => {
  // 实现...
}
```

它们作用不同，适用场景也不同。很多刚接触 TypeScript 的同学会疑惑：**“第二种不是更严谨吗？为什么实际代码中第一种出现得更频繁？”**

本文将深入对比这两种写法，并解释原因。

## 核心区别一览

| 对比维度 | 第一种（参数类型约束） | 第二种（`typeof` 完整类型） |
| :--- | :--- | :--- |
| **约束范围** | 仅约束参数 `option` 的类型 | 约束**整个函数**：参数 + 返回值 + 重载 + this 等 |
| **返回值类型** | 由函数体**自动推断**，可自由变化 | 必须与原函数的返回值类型**完全一致** |
| **灵活性** | 高：可改返回值、增加参数、变换行为 | 极低：任何偏差都会导致 TS 报错 |
| **类型依赖** | 只依赖 `Taro.downloadFile.Option` | 依赖完整的 `typeof Taro.downloadFile`（可能包含重载、静态属性等） |
| **错误提示** | 直接指出参数哪里不符合 `Option` | 可能抛出复杂的函数签名不匹配错误 |
| **适用场景** | **业务封装**（改变返回值、增加参数、添加逻辑） | **完美代理**（不改变接口行为，仅加日志/上报） |

## 为什么实际代码中常见第一种写法？

### 1. 封装函数通常会改变返回值类型

Taro 原生的 `downloadFile` 返回一个包含临时文件路径、状态码等信息的对象（`Promise<Taro.downloadFile.SuccessCallbackResult>`）。但在业务封装中，我们往往想直接返回文件内容（例如 `ArrayBuffer` 或 `string`），甚至直接返回下载后的本地路径字符串。

```typescript
// 业务封装：返回文件内容的 ArrayBuffer
export const downloadFileAsBuffer = async (option: Taro.downloadFile.Option) => {
  const res = await Taro.downloadFile(option)
  // 读取文件内容并返回 buffer
  return new Uint8Array(/* ... */)
}
```

此时返回值类型是 `Promise<Uint8Array>`，而 `typeof Taro.downloadFile` 的返回值是 `Promise<Taro.downloadFile.SuccessCallbackResult>`，两者不兼容。第二种写法就会报错，而第一种完美适配。

### 2. 避免引入不必要的复杂类型依赖

`typeof Taro.downloadFile` 可能会连带要求函数的 `this` 上下文、函数重载、甚至静态属性。如果你只是写一个简单的 `async` 函数，这些额外要求会带来类型错误。

```typescript
// 假设 Taro.downloadFile 有多个重载，以及需要特定 this 上下文
// 用 typeof 约束后，你的简单函数必须满足所有重载条件，非常痛苦
```

而第一种写法只使用了 `Taro.downloadFile.Option`，轻量、清晰，不会引入多余的类型约束。

### 3. 错误提示更友好

当你把参数写错时：

- **第一种**：TypeScript 会提示“类型 A 不能赋值给类型 `Taro.downloadFile.Option`，缺少属性 xxx”。
- **第二种**：TypeScript 可能会提示“不能将类型 `(option: ...) => ...` 赋给类型 `typeof Taro.downloadFile`”，并列出好几个重载签名不匹配，新手很容易被绕晕。

### 4. 允许增加额外的参数或配置

业务封装常需要增加自定义选项，比如重试次数、超时时间等：

```typescript
export const downloadFile = async (
  option: Taro.downloadFile.Option & { retryCount?: number }
) => {
  for (let i = 0; i < (option.retryCount || 1); i++) {
    try {
      return await Taro.downloadFile(option)
    } catch (e) { /* 重试 */ }
  }
}
```

第二种写法完全不允许你扩展参数，因为类型不匹配。

## 什么时候应该用第二种写法（`typeof`）？

第二种写法的真正价值在于 **“代理/包装原函数，但不改变任何接口行为”**。例如：

- 给原函数加一层统一日志 / 埋点。
- 临时 mock 原函数（返回假数据，但保持类型一致）。
- 对原函数进行错误上报，但仍透传返回值。

示例：

```typescript
// 完全代理，加日志，类型完全一致
export const downloadFile: typeof Taro.downloadFile = async (option) => {
  console.log('[Download]', option.url)
  return Taro.downloadFile(option)
}
```

此时你希望调用方完全感知不到区别，`typeof` 能确保你 100% 兼容原函数。

## 结论

- **第一种写法（参数类型约束）** 更常见，因为它**灵活、可扩展、错误提示友好**，符合“封装以适配业务”的需求。
- **第二种写法（`typeof` 复制类型）** 更严格，适用于需要**完全替代原函数且不改变接口**的场景。

两者没有绝对的对错，只有适用场景的不同。在封装第三方库时，优先考虑第一种——除非你明确需要第二种的严谨性。

> 记住：TypeScript 的类型系统是为了帮助我们写出更安全的代码，而不是束缚我们的创造力。根据实际需要选择合适的类型抽象，才是最佳实践。

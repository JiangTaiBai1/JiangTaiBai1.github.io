# 从"两道门"到线程池：内容生成系统的并发控制艺术

在 AI 内容生成这类高并发场景中，我们既要防止单个用户把系统资源占满，又要保证全局处理能力不被冲垮。本文基于一个真实的视频/图文生成系统，聊聊它如何用 **Redis + 线程池** 搭建两道可靠的"阀门"，并深入分析 **核心线程、队列、最大线程** 之间微妙的关系。

![线程图解](/images/posts/Thread-diagram.png)

## 一、两道门，两种职责

这个系统对每个用户有 **10 个** 并发任务的上限（视频和图文分别计算）。当用户提交任务时，需要连过两道门：

- **第一道门：Redis 用户级限流**  
  每个用户最多同时保有 10 个"进行中/排队中"的任务。超过 10 个直接拒绝，返回"当前排队任务已达上限"。

- **第二道门：全局线程池**  
  所有通过 Redis 检查的任务会进入同一个线程池，这里不再区分用户，完全按 **FIFO** 顺序执行。

这两道门一个管"谁能进来"，一个管"进来以后怎么排"，组合起来既保证公平又防止单用户霸占资源。

## 二、第一道门：Redis 用户任务计数

代码中使用了 Redis 的 ZSet 来计数：

```java
private static final String VIDEO_QUEUE_PREFIX = "video:queue:";
private static final int MAX_CONCURRENT_TASKS = 10;

public boolean canCreateVideoTask(String username) {
    String key = VIDEO_QUEUE_PREFIX + username;
    Long taskCount = redisTemplate.opsForZSet().count(key, 0, System.currentTimeMillis());
    return taskCount != null && taskCount < MAX_CONCURRENT_TASKS;
}
```

每个用户对应一个 ZSet key（如 `video:queue:张三`），任务提交时用时间戳作为 score 存入。`count` 方法统计当前仍在 ZSet 中的任务数，只要小于 10 就放行。任务执行完毕后，需要从 ZSet 中移除对应元素（例如通过任务标识或时间戳），从而"释放"一个名额。

这样做的好处是：**计数准确、原子操作、自带排序能力**。用户 A 提交 10 个任务后，第 11 个会直接触发 Redis 拦截，根本到不了线程池。而其他用户依然可以正常提交自己的 10 个任务。最终 3 个用户可能各自 10 个，共 30 个任务进入线程池。

## 三、第二道门：线程池的魔法——核心、队列、最大线程

通过 Redis 的任务会被提交给一个配置好的 `ThreadPoolExecutor`：

```java
executor.setCorePoolSize(10);      // 核心线程数
executor.setMaxPoolSize(50);       // 最大线程数
executor.setQueueCapacity(1000);   // 队列容量
executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
```

这个配置背后隐藏着线程池最核心的三个参数关系，我们用"餐厅厨房"来类比：

### 1. 核心线程（10 个）——"长期工厨师"

核心线程会一直存活（即使空闲也不会被回收，除非设置了 `allowCoreThreadTimeOut`）。任务一提交，如果当前线程数小于 10，就直接 **开启新线程执行**，不用排队。这 10 个线程就像餐厅的 10 个正式厨师，只要有活就马上干，没活也备着。

### 2. 队列（容量 1000）——"等位区"

当 10 个核心线程全在忙时，新任务会被放入一个等待队列（实际容量 1000）。这个队列相当于餐厅门口的等位区，任务按 FIFO 顺序等着。**队列里的任务不会分配新线程**，即使核心线程数没达到最大。这是 `ThreadPoolExecutor` 默认的行为：先塞满核心线程，然后进队列，队列满了才考虑创建非核心线程。

### 3. 最大线程数（50）——"临时工厨师"

只有 **队列也满了**（即 10 个核心线程全忙 + 1000 个任务排队），再来新任务时，线程池才会创建 **非核心线程** 来救急，直到线程总数达到 50。这就像等位区都坐满了，餐厅才会临时请外面的厨师来帮忙。非核心线程在空闲一段时间后会被回收。

### 4. 拒绝策略 ——"老板亲自上阵"

如果 50 个线程全忙，且队列也满（容量 1000），此时再提交任务就会触发 **拒绝策略**。这里配置的 `CallerRunsPolicy` 会让 **提交任务的线程自己执行这个任务**（比如 HTTP 请求线程直接跑 AI 生成任务）。这样不会丢失任务，但会阻塞调用方，相当于流量过大时老板亲自去端盘子，整个餐厅的出餐速度会急剧下降。

## 四、容量全景图

结合上面的参数，全局线程池可容纳的任务数计算如下：

- **直接执行**：最多 10 个（核心线程数）
- **队列排队**：最多 1000 个
- **救急线程执行**：当队列满时，最多再创建 40 个非核心线程（50 - 10），每个线程对应一个任务执行（队列已满，新任务直接被线程领取）

所以系统总共可以"收纳"的任务量为：

> **10（执行中） + 1000（排队） + 40（扩容执行中） = 1050 个**

第 1051 个任务会触发 `CallerRunsPolicy`，由调用者线程同步执行。

注意：当任务在扩容线程中执行时，队列依旧是满的，因为这些任务没有经过排队，是直接交给新线程的。一旦有线程空闲出来，就会立刻从队列头部拿一个任务继续处理。

## 五、实际运行场景举例

假设有 3 个用户各提交 10 个视频生成任务，共 30 个任务通过 Redis 到达线程池：

1. 前 10 个任务直接被 10 个核心线程领走执行。
2. 剩余 20 个任务进入队列等待。
3. 假设每个任务耗时不同，核心线程陆续空闲，从队列里取走下一个任务。这 30 个任务完全 **混在一起** 按提交时间顺序执行，与用户无关。

此时如果突然来了一个"大客户"一次性提交 500 个任务，而 Redis 将其放行（因为这是他今天的前 500 个，且他之前没有任何排队），那么：

- 10 个核心线程 + 队列（1000 容量）可以轻松容纳这 500 个任务，全部排队慢慢执行，不会触发扩容。

只有当同时堆积的任务超过 1010 个时，才会开始创建第 11 个线程……直到达到 50 线程 + 满队列，才开始触发拒绝。

## 六、Redis 与线程池的协同

这个系统最精妙的地方在于两级控制：

- **Redis 保证用户公平**：任何用户最多占用 10 个并发名额，避免了某个用户提交几千个任务把队列占满，从而影响其他人。
- **线程池保证全局吞吐**：核心线程数固定，队列容量有限，最大线程可控，防止系统资源耗尽。`CallerRunsPolicy` 确保任务不丢失，起到"背压"作用。

因此，30 个任务并不会触发线程池的扩容，它们安静地在队列里等待 10 个核心线程轮番处理。只有当短时间真有超过 1010 个任务同时涌入时，线程池才会逐步扩容到 50，给系统一个缓冲期；实在处理不过来，就让调用方慢下来，这就是一种优雅的降级。

## 七、注意事项

- **Redis ZSet 的清理**：任务完成或失败时，必须从 ZSet 中移除对应条目，否则用户计数器会一直增长，导致无法提交新任务。可以用时间戳作为 member 精确删除，或者在任务结束时按 score 范围删除。
- **队列容量不宜盲目加大**：1000 的队列已经很大，如果系统经常接近满队列，说明核心线程数或处理速度跟不上，应考虑优化任务执行逻辑，而不仅仅增大队列，否则会增加延迟。
- **拒绝策略的副作用**：`CallerRunsPolicy` 会让 HTTP 线程阻塞，如果大量任务被拒绝，会导致 Web 接口无响应。实际中可结合监控，动态调整参数或接入消息队列削峰。

## 总结

- **核心线程** = 常驻干活的人，优先处理，不够就排队。
- **队列** = 等位区，能让系统在高峰期扛住压力，但会延迟。
- **最大线程** = 临时救急的人，队列满了才出场。
- **Redis 层** = 每个用户的"准入证"，防止一个人占满全局队列。

这种双层设计既保证了用户体验（每人有自己的并发上限），又保证了服务端的稳定性（全局线程池可控），是内容生成类系统的经典架构。希望这篇文章能帮你理清线程池参数之间的关系，下次再看到 `corePoolSize`、`queueCapacity`、`maxPoolSize` 时，脑海里能马上浮现出那个"厨房 + 等位区 + 临时工"的画面。
